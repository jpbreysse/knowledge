# Plan: v2.1 — Lummus due diligence overlay

## Context

We keep the existing asset data (cooling water utility + KM-145 pumping station) as the demo plant. This increment adds a **due-diligence overlay**: one or more Lummus assessments attached to an asset, each with a condition rating, remaining useful life, findings, recommendations, and CapEx estimate. It turns the plain asset register into a customer-facing DD portal.

The user is a customer plant engineer. The portal is **read-only** — assessments come from the seed file, there is no authoring UI in v2.1. Writing is a later concern once Lummus wants to internally author assessments in-app.

v2.0 typed-relationships stays on hold until it directly strengthens the pitch.

None.

## Deliverables

### 1. Migration: `migrations/004_asset_assessment.sql`

Plain SQL, wrapped in a transaction.

```sql
CREATE TABLE asset_assessment (
  id                            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  asset_id                      UUID NOT NULL REFERENCES asset(id) ON DELETE CASCADE,
  condition_rating              TEXT NOT NULL
    CHECK (condition_rating IN ('A','B','C','D')),
  remaining_useful_life_years   NUMERIC(5,1),
  risk_score                    SMALLINT CHECK (risk_score BETWEEN 1 AND 5),
  capex_estimate_usd            BIGINT,
  status                        TEXT NOT NULL DEFAULT 'assessed'
    CHECK (status IN ('assessed','pending','not_in_scope')),
  assessed_by                   TEXT,
  assessed_on                   DATE,
  summary                       TEXT,
  findings                      JSONB NOT NULL DEFAULT '[]'::jsonb,
  recommendations               JSONB NOT NULL DEFAULT '[]'::jsonb,
  created_at                    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_asset_assessment_asset_id  ON asset_assessment (asset_id);
CREATE INDEX idx_asset_assessment_condition ON asset_assessment (condition_rating);
CREATE INDEX idx_asset_assessment_status    ON asset_assessment (status);
```

**Shape rules:**
- `findings`: JSONB array of `{ severity: 'critical'|'major'|'minor'|'observation', title, detail }`.
- `recommendations`: JSONB array of `{ action, priority: 'urgent'|'planned'|'monitor', capex_estimate_usd, timing }`.
- Multiple rows per asset allowed (reassessments over time). The UI shows the **latest** one by `assessed_on` desc; older ones are kept for future history-tab work.
- `condition_rating` meanings (documented): A = excellent / like new, B = good / minor wear, C = needs attention / planned refurbishment, D = critical / immediate action.
- `risk_score` is forward-looking failure risk (1 low → 5 high), distinct from existing `asset.criticality` which is static operational importance.

### 4. Read-only API

One endpoint, since the demo doesn't author assessments:

- `GET /api/assets/[id]/assessments` — all assessments for the asset, newest first. Returns the raw rows; the detail page picks the latest.

The **list** query (`GET /api/assets`) and the **graph** query (`GET /api/graph`) both need to expose the latest-condition per asset so list/graph UIs can render without a second round-trip. Add a subquery that joins the most recent `asset_assessment` per `asset_id` (using a `LATERAL` join or a window function). Return `latest_condition` and `latest_assessment_status` as extra fields on every asset row.

**Still one query per page** — no N+1.

### 5. Asset detail page (`/assets/[id]`)

Insert a new tab **"Lummus DD"** between "Details" and "Documents". Tab contents, pulled from the latest assessment:

- **Header strip**: large condition badge (A/B/C/D with colour), RUL years, risk score, CapEx total, status, assessed date + assessor.
- **Summary**: the `summary` text, styled as a blockquote.
- **Findings**: list grouped by severity (critical first). Each row: severity badge + title + detail.
- **Recommendations**: table with columns `Action`, `Priority`, `Est. CapEx`, `Timing`.
- **Empty state** (no assessment row, or `status = 'not_in_scope'`): a neutral box saying "No Lummus assessment for this asset yet" or "This asset was not in scope of the recent DD engagement".

## Scope boundaries — do NOT build

- **No write API** (no POST/PATCH/DELETE on assessments). The demo is read-only.
- **No in-app authoring UI** (finding editor, recommendation editor).
- **No PDF report export** of the DD tab.
- **No customer branding / logo / colour palette** — separate task.
- **No multi-tenancy, no auth.**
- **No embeddings or RAG on findings.**
- **No history tab for multiple assessments** — the schema supports it, the UI doesn't yet.
- **No CapEx roll-ups by system/area** — four global KPI tiles, nothing deeper.
- **No notifications** when a critical finding is added.
- **No changes to typed relationships** — v2.0 stays on hold.
