# Plan: v3.3 — Rules → Proposals (the Logic layer)

## Context

The v3.x builds so far have delivered the **Data** layer: assets, classes, entity_links, attributes JSONB, versioning, DD assessments, findings-panel readback from the Knowledge app. What's missing to earn the "ArborSpace = Palantir but delivered" positioning is the **Logic** layer — declarative rules that reason over the register and emit **proposals** (findings with `type='proposal'`, `status='raised'`) into the Knowledge app inbox.

Palantir's frame is Data → Logic → Actions. Their AI doesn't act directly: it proposes, a human confirms. That's the exact loop this spec builds. It also uses each of the three apps in its natural role:

| App | Owns |
|---|---|
| **Asset app (5177)** | **Rules** — declarative policies attached to asset classes. Evaluator emits proposals on write. |
| **Knowledge app (5178)** | **Findings** including proposals. Type vocabulary extends from `inspection` → `inspection` / `proposal` / `rule_derived`. Existing list becomes the proposals inbox. |
| **Document app (3001/5173)** | Unchanged — description docs behind each finding. |

**Zero new domain concepts.** No new tables in the Knowledge app; only a widened CHECK. The existing `PrecedentsPanel` (v2.3) already implements "what have we done in similar situations" — that closes the loop.

## Design decisions locked

1. **Rule condition language**: **[JSONLogic](https://jsonlogic.com/)** — well-tested, tiny (~5 KB), JSON-shaped (edits cleanly via the admin UI without a parser), evaluates identically on server + client. Reject JSONata (heavier), reject SQL fragments (limited to columns), reject arbitrary JS (unsafe).
2. **Evaluator trigger**: fires **on asset write** in the API handler (POST/PATCH), after the row is committed. Plus a **"Run rules against register" button** for backfill / after adding a new rule.
3. **Cross-app call**: uses the existing `connector` registry (v2.2). One new seeded connector `knowledge-app`. The asset app POSTs proposals to `<knowledge>/api/findings`.
4. **Idempotency**: `(rule_id, target_asset_id, kind='rule')` is unique in a new `rule_emission` ledger on the asset side. Re-evaluating an already-fired rule doesn't double-emit.
5. **No automated side effects on accept in v3.3.** Accepting a proposal in the Knowledge app changes its status; the human then makes the change (or spawns a follow-up finding). Side-effect callbacks are v3.4.

## Deliverables

### P1 — Asset app: `asset_rule` table + Drizzle mirror · ~1.5 h

Migration `migrations/014_asset_rule.sql`:

```sql
CREATE TABLE asset_rule (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code          TEXT NOT NULL UNIQUE
    CHECK (code ~ '^[a-z][a-z0-9-]{2,49}$'),
  label         TEXT NOT NULL,
  class_code    TEXT,                 -- if set, rule only evaluates against assets of this class
  condition     JSONB NOT NULL DEFAULT '{}'::jsonb,     -- JSONLogic expression
  emit_type     TEXT NOT NULL DEFAULT 'proposal'
    CHECK (emit_type IN ('proposal', 'rule_derived')),
  emit_severity TEXT NOT NULL
    CHECK (emit_severity IN ('low','medium','high','critical')),
  emit_title    TEXT NOT NULL,        -- template with {tag}, {name}, {attributes.foo}
  emit_body     TEXT,                 -- optional richer description template
  enabled       BOOLEAN NOT NULL DEFAULT TRUE,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_asset_rule_enabled ON asset_rule (enabled) WHERE enabled;
CREATE INDEX idx_asset_rule_class   ON asset_rule (class_code) WHERE class_code IS NOT NULL;
CREATE TRIGGER trg_asset_rule_set_updated_at
  BEFORE UPDATE ON asset_rule FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- Idempotency ledger — one row per (rule, asset) firing.
CREATE TABLE rule_emission (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rule_id         UUID NOT NULL REFERENCES asset_rule(id) ON DELETE CASCADE,
  asset_id        UUID NOT NULL REFERENCES asset(id) ON DELETE CASCADE,
  finding_id      TEXT,               -- external ref into kn_findings.id
  finding_url     TEXT,
  emitted_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (rule_id, asset_id)
);
CREATE INDEX idx_rule_emission_rule ON rule_emission (rule_id);
CREATE INDEX idx_rule_emission_asset ON rule_emission (asset_id);
```

Drizzle mirror in `src/lib/server/db/schema.ts`. Types + read helpers in `src/lib/server/asset-rules.ts`.

### P2 — Asset app: JSONLogic evaluator + emitter · ~2.5 h

- `npm install json-logic-js` — 5 KB.
- New `src/lib/server/rule-evaluator.ts`:
  - `evaluateRulesForAsset(assetId)` — loads rules where `enabled AND (class_code IS NULL OR class_code = asset.class_code)`, evaluates each `condition` against a normalised asset object `{tag, name, classCode, attributes, confidentiality, version, parentId, ...}`, returns an array of matches with the rendered title.
  - Template rendering: simple `{tag}` / `{attributes.criticality}` substitution — one small function, no dep.
- New `src/lib/server/rule-emitter.ts`:
  - `emitProposal(rule, asset)` — POSTs to the knowledge-app connector's `/api/findings` endpoint with `{title, severity, type: emit_type, description_html, asset_refs: [{asset_id, asset_tag, asset_display}]}`. Records the response `id` + URL into `rule_emission`. Silently skips if the `(rule, asset)` pair already exists (idempotency).
- Wire into POST `/api/assets/+server.ts` and PATCH `/api/assets/[id]/+server.ts` — after the successful INSERT/UPDATE, `await evaluateRulesForAsset(row.id)` then `emitProposal(...)` per match. Failures logged, don't block the request.

### P3 — Asset app: `/asset-rules` admin UI · ~2.5 h

Mirrors the `/asset-types` pattern:
- List page — code, label, class_code, severity, enabled, actions.
- New / edit pages — a form with:
  - identity: code, label, class_code dropdown (from `asset_class`)
  - condition editor: for v3.3, a **raw JSON textarea** with 3–4 preset examples in a sidebar (`criticality >= 4`, `attributes.status == "in DD" AND attributes.debt_amount_usd_m > 500`, etc.). A structured builder is v3.4.
  - emit fields: severity dropdown, type dropdown, title template, body template
- API: `/api/asset-rules` GET/POST, `/api/asset-rules/[id]` GET/PATCH/DELETE.
- **"Run rules against register"** button on the list page — POSTs to `/api/asset-rules/run-all`, which iterates every enabled rule × every matching asset, calls `emitProposal` for each. Progress toast (`Fired 12 proposals across 47 assets`).

## Rule shape — worked example

```json
{
  "code": "low-rul-critical",
  "label": "Low remaining useful life — critical",
  "class_code": "PUMP-CENTRIFUGAL",
  "condition": {
    "<": [{"var": "attributes.remaining_useful_life_years"}, 1]
  },
  "emit_type": "proposal",
  "emit_severity": "critical",
  "emit_title": "Immediate refurbishment recommended: {tag} shows RUL {attributes.remaining_useful_life_years} year(s)",
  "emit_body": "Rule {code} matched. Suggested action: schedule a refurbishment window within Q3."
}
```

## Data flow — worked example

1. User updates `P-305.attributes.remaining_useful_life_years` from 0.5 to 0.3 via the AssetForm.
2. PATCH `/api/assets/<P-305 id>` — validation + update commits.
3. Handler calls `evaluateRulesForAsset(P-305.id)`. `low-rul-critical` matches.
4. Handler calls `emitProposal(rule, asset)`.
5. `rule-emitter` looks up `(rule.id, asset.id)` in `rule_emission` — no row → proceed.
6. Reads the `knowledge-app` connector. POSTs to `http://localhost:5178/api/findings`:
   ```json
   {
     "title": "Immediate refurbishment recommended: P-305 shows RUL 0.3 year(s)",
     "severity": "critical",
     "finding_type": "proposal",
     "description_html": "<p>Rule low-rul-critical matched…</p>",
     "asset_refs": [{ "asset_id": "<P-305 uuid>", "asset_tag": "P-305", "asset_display": "P-305 — Sump drain pump" }]
   }
   ```
7. Knowledge app inserts a `kn_findings` row + `kn_finding_asset` link.
8. Response body has `{id, url}`. Asset app inserts a `rule_emission` row.
9. Detail page for P-305 refreshes — the existing FindingsPanel (v2.3) pulls this finding since it's now linked. Also visible on `/proposals`.

## Out of scope (v3.4+)

- **Cross-class conditions** — "engagement has no IER yet". Requires querying beyond the single asset the rule fires on. Doable via JSONLogic + a small `related(asset, classCode)` helper injected as `var:_ctx.related.IER.length == 0`, but complexity budget for v3.3 doesn't cover it.
- **Automated side effects on accept** — accepting a proposal today only changes status. v3.4 could POST back to the asset app to apply a change (e.g. flip `confidentiality` to `restricted`).
- **Rule versioning** — rules aren't versioned. When you edit a rule, past emissions stay linked to the (now-different) rule row.
- **Bulk accept / reject** from the Proposals page — one at a time in v3.3.
- **Scheduled evaluation** (cron) — rules only re-fire on asset write. If no asset changes, no proposals appear even if the rule was just enabled — that's what the "Run against register" button covers.

## Positioning payoff

Once this ships, the pitch line lands cleanly without any qualifiers:

> **"Palantir's ontology is the right idea. We deliver it as a finished tool for one team — on your own servers — with the rules for DD already wired between the register, the rules engine, and the findings inbox. The category leader spent billions proving the shape. We ship it Tuesday."**

Because:
- **Data**: the asset register with class-driven attributes, versioning, entity_links — already shipped.
- **Logic**: `asset_rule` + evaluator — this spec.
- **Actions**: proposals in the Knowledge app inbox, human accept/reject, precedents feed forward — v2.3 + this spec.

The three-app HTTP-federated architecture is preserved end-to-end. No shared FKs. No shared codebase. Each app does the thing it already does; the vocabulary just extends where the story asked it to.
