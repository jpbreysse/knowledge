# Plan: v3.0 — LCI Asset Register schema

## Context

The asset app at `/Users/jean-philippebreysse/dev/asset` already hosts the **Lummus industrial-equipment demo** (pumps, motors, pig traps, meters under `EQUIPMENT` family) and a placeholder `LENDER-CONTEXT` entry. This increment adds a **second, fundamentally different vertical**: the **LCI Independent Engineer asset register** — 17 asset types organised in a three-layer ontology (Client / Engineer / Company), drawn from the user-supplied `Asset_Definitions.docx` spec.

**Decision (confirmed)**: the two verticals coexist in the same `asset_class` taxonomy. Lummus classes keep `family='EQUIPMENT'/'STRUCTURAL'/...`; LCI classes get new families `LAYER-1`/`LAYER-2`/`LAYER-3`. The list/graph/detail pages already filter by family.

## What fits today (no new code)

About 70% of the LCI schema slots cleanly into the existing data-driven taxonomy. Each LCI class becomes an `asset_class` row whose `attributeFields` declares its fields using the existing `text` / `number` / `select` / `boolean` / `list` / `object` types. Per-class lifecycle and tab visibility already work.

## What's missing — 4 capability gaps

### Gap 1 — Cross-asset reference fields (the unblocker)

The LCI spec is full of `ref` and `array[ref]` fields: `sponsor_ref → spon-*`, `syndicate_members[] → lend-*`, `physical_asset_ref → phys-*`, `owner_ref → eng-*`, etc.

**Build**: two new field types — `ref` (single) and `array_ref` (many). Each field def gains an optional `asset_class_filter` (or array) so the picker is scoped to the right class. The dynamic form renderer gets an asset-picker widget (typeahead by tag/name). Validation: each ref must resolve to an existing asset of the right class. Storage stays in the existing `attributes` JSONB as a UUID string or array of UUIDs.

### Gap 2 — Layer vocabulary + `id_prefix`

- **Extend the `asset_class.family` CHECK** to include `LAYER-1`, `LAYER-2`, `LAYER-3` alongside the existing 4 families.
- **Add `id_prefix` TEXT** to `asset_class` (e.g. `'phys-'`). On asset create, pre-fill `tag` as `<prefix>NNN` where NNN is the next sequential number for that class.

### Gap 3 — Confidentiality

The LCI spec defines `public` / `client-shareable` / `internal` / `restricted` and a default per class.

- **`asset_class.confidentiality_default`** — new column, nullable.
- **`asset.confidentiality`** — new column, nullable. Falls back to the class default. Rendered as a badge in the header; filterable on the list page.
- **No access enforcement** — this is signalling, not auth (still single-user demo).

### Gap 4 — Versioning (IER, methodology, lender-playbook)

- **`asset.version`** — free-form TEXT (e.g. `"v0.3-draft"`).
- **`asset.supersedes_asset_id`** — UUID self-ref, `ON DELETE SET NULL`. Detail page shows the chain.

## Coexistence rules

- Lummus classes keep their existing `family` values and `EQUIPMENT`-style behaviour.
- LCI classes use `LAYER-1`/`LAYER-2`/`LAYER-3`.
- The IDB Invest seed row (`lender-ctx-001`) gets re-mapped to the new richer LCI `LEND` class (extra fields appear blank initially — backwards-compatible).
- Existing `applicable_tabs` semantics carry over: the Lummus DD tab stays hidden for `LAYER-1` classes (none of them want it).

## Out of scope

- **No engagement table.** `engagement_ref` becomes a free-form text/URL field for now.
- **No finding table in the asset app.** The existing `kn_findings` (Knowledge app at :5178) plus the FindingsPanel/PrecedentsPanel continue to serve this need.
- **No real auth / per-confidentiality enforcement.** The badge is informational.
- **No multi-tenant sequence allocation.** `tag` auto-prefix uses `SELECT MAX(seq)+1` — fine for single-user demo, race-condition-tolerant via a DB sequence later.
