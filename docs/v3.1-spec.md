# Plan: v3.1 — DD demo vertical + typed cross-links

## Context

The v3.0 build shipped an LCI Independent-Engineer register (17 classes, Peru-pipeline demo seed at [seed/011_lci_demo.sql](../seed/011_lci_demo.sql)). This increment adds a **second demo vertical** — a small Lummus DD dataset with 4 engagements — and lands the deferred v2.0 typed-relationships work (originally scoped in [docs/v2-spec.md](v2-spec.md) as `asset_relationship`; the new spec calls the same idea `entity_link`).

The point of the new vertical is to demo **cross-engagement retrieval** — click a lender/sponsor/advisor in the graph, see every deal they've participated in — which is not achievable with the single-parent `parent_id` tree alone.

## Decisions locked (from spec review)

1. **Naming**: class code `ENGAGEMENT` (avoiding collision with the existing `ENG` = Engineer class), id_prefix `engmt-`, tags `engmt-001..004`.
2. **Families**: migrate. The 5 classes that participate in this demo move to the new family scheme:
   - `ENGAGEMENT` → new family `PROJECT`
   - `SPON`, `LEND`, `EXTADV` → new family `PARTY` (currently `LAYER-1`)
   - `PHYS` → new family `ASSET` (currently `LAYER-3`)
   - Other LCI classes (METH, RFI, COMP, COMM, COMMP, SITE-VISIT, PLAY, INDIV, TX, IER, CMR, ENG, ENGH) keep their `LAYER-1/2/3` families — untouched.
3. **Coexistence**: existing Peru-pipeline seed (011) stays intact. New DD demo uses distinct tag numbers (`spon-003..005`, `lend-003..005`, `extadv-002..004`, `phys-002..003`) — no tag collisions, no re-seed of 011.

### P2 — `entity_link` table + Drizzle mirror · ~1.5 h

Migration `migrations/012_entity_link.sql`:

```sql
CREATE TABLE entity_link (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_id     UUID NOT NULL REFERENCES asset(id) ON DELETE CASCADE,
  target_id     UUID NOT NULL REFERENCES asset(id) ON DELETE CASCADE,
  relation_type TEXT NOT NULL CHECK (relation_type IN ('participates_in')),
  attributes    JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_by    TEXT,
  CHECK (source_id <> target_id),
  UNIQUE (source_id, target_id, relation_type)
);
CREATE INDEX idx_entity_link_source ON entity_link (source_id);
CREATE INDEX idx_entity_link_target ON entity_link (target_id);
CREATE INDEX idx_entity_link_type   ON entity_link (relation_type);
```

Naming note: adopted the doc's `entity_link` / `source_id`+`target_id` / `relation_type` vocabulary. This is the same shape as the original v2.0 `asset_relationship` spec — replaces it.

Vocabulary for v3.1: `participates_in` only. `secondary_advisor` and future types added by a small `DROP CONSTRAINT / ADD CONSTRAINT` migration when needed.

Drizzle mirror in `src/lib/server/db/schema.ts`.

## Attribute schema for `ENGAGEMENT`

```
name            (text, required)   — Engagement title
engagement_type (select, required) — "Environmental & Decommissioning" | "Lender's Engineer" |
                                     "M&A Technical DD" | "Lender-Side Asset Valuation" |
                                     "Construction Monitoring" | "other"
mandate_start   (text)             — ISO date
mandate_end     (text)             — ISO date, nullable
status          (select, required) — "in DD" | "delivered" | "on hold" | "closed-out"
```

Confidentiality default: `restricted`. Applicable tabs: `["details","documents","history"]`.
